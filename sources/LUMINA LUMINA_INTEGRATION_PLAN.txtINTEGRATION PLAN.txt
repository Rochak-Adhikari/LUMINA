INTEGRATION PRIORITIES (in order)
--------------------------------------------------------------------------------

PRIORITY 1 — file_processor (LOW EFFORT, HIGH VALUE)
  Source: Mark-XLVI/actions/file_processor.py
  What it adds: AI analysis of uploaded files — PDFs, images, CSV, audio, video.
  Lumina currently has file_controller.py for CRUD but NOT content analysis.
  Port to: backend/actions/file_processor.py
  Register in ACTION_REGISTRY using existing @register_action pattern.
  Hook file upload into frontend (AppTest.jsx drag-and-drop or file input).

PRIORITY 2 — dev_agent (MEDIUM EFFORT, HIGH VALUE)
  Source: Mark-XLVI/actions/dev_agent.py
  What it adds: Full multi-file project scaffolding. Plans, writes every file,
  installs dependencies, opens VS Code, runs code, auto-fixes errors.
  Lumina has code_helper.py for single snippets only.
  Port to: backend/actions/dev_agent.py
  Note: dev_agent calls code_helper internally — keep both.

PRIORITY 3 — Remote Phone Dashboard (MEDIUM EFFORT, UNIQUE FEATURE)
  Source: Mark-XLVI/dashboard/
  What it adds: Control Lumina from your phone via LAN browser.
  View status, send text commands, trigger voice — all from phone.
  Wire into: Lumina's existing FastAPI server (server.py HTTP routes).
  Do NOT replace the Electron frontend. This is an additional mobile interface.

PRIORITY 4 — flight_finder (LOW EFFORT)
  Source: Mark-XLVI/actions/flight_finder.py
  What it adds: Google Flights search via web scraping.
  Port to: backend/actions/flight_finder.py

PRIORITY 5 — agent_task (HIGH EFFORT, HIGH VALUE)
  Source: Mark-XLVI/actions/agent_task.py
  What it adds: Multi-step autonomous task execution with goal + priority queue.
  Lumina routes through Gemini tool calls but has no persistent task queue.
  Port to: backend/actions/agent_task.py
  Needs: a task queue (asyncio.Queue or similar) wired into server.py.

PRIORITY 6 — game_updater (LOW EFFORT — optional)
  Source: Mark-XLVI/actions/game_updater.py
  What it adds: Steam/Epic Games install and update control.
  Port only if gaming is a use case.

--------------------------------------------------------------------------------
RULES FOR ALL INTEGRATION WORK
--------------------------------------------------------------------------------
1. NEVER touch the persona engine without explicit instruction. It is Lumina's
   core identity. Changes to persona_engine.py require separate approval.

2. NEVER remove or bypass the permission/confirmation gate. All destructive tool
   actions require explicit user confirmation. No exceptions.

3. NEVER delete or reformat memories in lumina_memory.db. User memories are
   sacred. Any migration must be additive only.

4. ALWAYS register new action modules in ACTION_REGISTRY using the existing
   @register_action decorator pattern. Do not add bare if/elif to action_router.py.

5. ALWAYS keep Gemini Live as the voice pipeline. Claude and other models are
   for reasoning/code/analysis tasks only. Never replace real-time voice.

6. ALWAYS test against the existing test files after changes:
   backend/test_phase_b1.py
   backend/test_phase_c.py
   backend/test_phase_e3.py
   backend/test_memory_suggest.py

7. When porting from Mark-XLVI, adapt the code to Lumina's patterns:
   - Use Lumina's ACTION_REGISTRY decorator, not Mark's dispatch table
   - Use Lumina's memory_store API (add_memory, get_memories), not Mark's JSON
   - Use Lumina's async patterns (lumina.py is fully async)
   - Preserve all existing return string format (Lumina speaks responses aloud)

8. Any new .env variables must be added to BOTH .env.example AND documented here.
